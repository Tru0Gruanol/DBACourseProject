## 二、修复内容

### 2.1 删除操作的关联检查

在 `StudentEnrollmentMapper.java` 中新增了两个计数方法：
- `countByClassCode`：统计某班级下有多少学生报名
- `countByStudentId`：统计某学生选了多少课

在 `AccountMapper.java` 中新增了：
- `countByStudentId`：统计某学生有多少条账目流水

然后修改了 `ClassesService.deleteClass` 和 `StudentService.deleteStudent`，在执行 DELETE 之前先调用这些计数方法检查，如果有关联数据就阻止删除并给出明确的提示。

### 2.2 enrolledCount 字段的双重防护

**后端**：`ClassesService.updateClass` 在更新前先从数据库查出当前的 enrolledCount，然后把前端表单传过来的值强制覆盖掉——也就是说，不管你在表单里填什么数字，最后写入数据库的 enrolledCount 始终是真实值。

**前端**：`ClassManage.vue` 编辑弹窗中，enrolledCount 输入框在编辑模式下变为禁用状态（灰色），旁边加了提示文字"由系统自动维护，不可手动修改"。

### 2.3 "假删除成功"Bug 的修复（重点）

这是今天发现的最隐蔽的 Bug。

**问题现象**：网页上做任何删除操作，都能弹出"删除成功"的绿色提示，但其实数据库里的数据完全没动。

**根因**：后端的删除方法在遇到失败时（比如"科目下面还有班级"），返回的是字符串 `"删除失败：xxx"`，但 HTTP 状态码是 200（因为后端确实收到了请求并给出了回复，不算网络层面的错误）。前端的 Axios 拦截器看到 HTTP 200 就当成功处理了，各个页面组件在 `await deleteXxx()` 之后，不检查返回内容就直接显示绿色的"删除成功"，所以不管后端返回的是"成功"还是"失败"，前端一律显示绿色。

**修复方式**：在 `request.js` 的响应拦截器里加了一层字符串内容检测——如果后端返回的字符串里包含"失败"、"已存在"、"不能"这些字眼，就直接当错误处理，弹出红色错误提示，并且不让后续的刷新数据代码继续执行。同时给缺少 try/catch 包裹的 `SubjectManage.vue` 的保存函数补上了异常处理。

### 2.4 其他修复

- **ScheduleQuery.vue**：学生课表标签页现在会加载教师列表，把教师 ID 映射成中文姓名显示
- **FeeManage.vue**：缴费金额加了独立的 `> 0` 校验
- **StudentEnrollment.vue**：报名成功后保留科目选择，自动重新拉取班级列表刷新名额
- **request.js**：错误拦截器现在能正确处理四种格式的后端响应（纯字符串、`{message}`、`{error}`、裸对象）
- **6 个 Entity 类**：全部改成 Lombok `@Data` 注解，删掉了约 280 行手写 getter/setter
- **新增 `GlobalExceptionHandler.java`**：全局异常处理器，统一捕获数据库约束冲突、参数缺失等异常
- **新增 `dto/Result.java`**：统一 API 响应模板，为后续接口标准化做准备

### 2.5 关于"数据到底存在哪"的误区

我之前一直以为网页上改了数据，数据会写回 `insert_test_data.sql` 文件，所以在 MySQL Workbench 里打开那个 SQL 文件看，发现从来没变过，就以为数据没存上。

实际情况是：`insert_test_data.sql` 只是一个一次性的初始化脚本，执行完就"退休"了。网页上所有的增删改操作，是直接发生在 MySQL 数据库 `tutoring_center` 的表里。要想看数据变化，应该在 MySQL Workbench 里执行 `SELECT * FROM 表名;`，而不是打开 SQL 文件。

## 四、发现的新问题：删除逻辑中的"踢皮球"死锁

今天在修复删除操作的过程中，我发现了一个更深的逻辑漏洞，这个问题今天没有解决，准备明天专门思考后再动手。

### 问题描述

目前后端的账目逻辑只考虑了一种情况：**学生欠费**（已缴 < 学费）。催费列表也是基于这个前提设计的。

但如果学生**多缴了钱**（已缴 > 学费），比如学费 2800 但缴了 3000，系统完全没有"退款"这个概念。这就导致了一个"踢皮球"式的死锁链：

```
想删除一个学生？
  → 不行，因为 student_enrollments 表里有他的报名记录
      → 想删除报名记录？
          → 不行，因为 accounts 表里有他的缴费流水
              → 想删除缴费流水？
                  → 不行，因为金额对不上（多缴了 200，没地方退）
                      → 想退钱？
                          → 系统根本没有退款功能！
                              
        于是：谁都动不了，死锁了。
```

**每一环都锁着下一环，最后卡死在"没有退款机制"上。**

### 问题的根源

账目模块的设计只考虑了"收钱"这一条路径：
- 报名时缴费 ✓
- 单独补缴 ✓
- 查询欠费（`amount_paid < fee`）✓

但完全没有考虑"退钱"的路径：
- 多缴了怎么办？ ✗
- 退课时已交的钱怎么处理？ ✗
- 多缴的余额怎么清算？ ✗

### 明日计划

这个问题不是改一两行代码能解决的，涉及到：
- 数据库表结构是否需要调整
- 退款业务流程怎么设计
- 删除学生/班级/报名时的资金清算逻辑
- 前端收费管理页面是否需要新增退款功能

准备明天专门花时间深度思考后，给出一个完整的方案再动手修改。
